MC Waterslides — Architecture & Delivery Plan

v1 planning doc · 2026-07-06 · PGMacDesign · repo: PGMacDesign/MCWaterSlides (public, MIT)

1. Repo & stack

The repo deliberately mirrors PGMacDesign/MC3DPrint (post-multi-version merge) in structure, build setup, CI, and conventions. Where this doc is silent, MC3DPrint is the answer.

  • Java 21, NeoForge, official Mojang mappings, Gradle + ModDevGradle + Stonecutter (Groovy controller).
  • mod id mcwaterslides · package com.pgmacdesign.mcwaterslides · display name MC Waterslides.
  • Multi-version from day one: one tree, one jar per NeoForge node — 1.21.1 · 1.21.8 · 1.21.9 · 1.21.10 · 1.21.11 · 26.1 · 26.2 (extend as new MC versions ship). 1.21.1 is the base/vcs node: plain code targets it; other nodes via Stonecutter guards/replacements. Java 21 launcher JVM for the daemon; 26.x nodes compile on a foojay-provisioned Java 25 toolchain.
  • Sequencing: scaffold all seven nodes up front (compile-green stubs), but make 1.21.1 fully fun first; walk the forward ladder green as a dedicated ship milestone rather than porting every feature seven times mid-flight.

3. The two hard problems (spike early)

These two carry the project risk. Both get spiked inside M1, before anything else is polished. Run derive-invariants on the momentum/current tickets before implementation and write the Invariants section back to the ticket.

3a. High-speed player movement vs. the server

At 22 b/s the vanilla "player moved too quickly" check and client/server desync are the #1 jank source. Approach:

  • Client prediction for the local player: the same momentum integration runs client-side while riding; the server stays authoritative and reconciles. Riding state + momentum sync via a small payload.
  • Movement-check mitigation: flag actively-riding players so the speed check tolerates slide velocity (bounded by speedCap — the cap exists partly for this), restore vanilla checks the tick riding ends.
  • Fallback if playtests still show jank at cap: the hybrid seat-entity above a speed threshold (pre-agreed plan B; design doc §2).

3b. Current propagation cost

Naive per-tick BFS through water from every jet is O(range × cross-section) × jets. Approach: compute each jet's current field once on placement/power-on, cache it, and invalidate on block updates within the field's bounds; hard cap field volume per jet; riders sample the cached field (a lookup, not a search). maxPushedEntitiesPerJet bounds per-tick entity work on mega-parks.

4. Testing strategy

  • Mobs make physics testable. Because mobs ride (design §4), GameTests can spawn a villager in a channel and assert real physics without simulating a player: current_pushes_mob_along_channel, uphill_jet_break_even, no_power_no_push, item_transport, splash_pool_stops_rider, flood_valve_fills_enclosure (+ leak detection), channel_autoconnect_shapes.
  • JUnit for pure logic: momentum math (drag, slope exchange, cap clamp), current-field propagation/invalidation, flood-fill bounds.
  • MC3DPrint's hard-won rules apply verbatim: neoforge.enabledGameTestNamespaces on all three run configs (else zero gametests register and the run false-greens); test workingDir pinned to root; GameTests live on the 1.21.1 oracle node, forward nodes get compile + boot-smoke.

5. Multi-version notes

  • Highest-churn areas across 1.21.x → 26.x for this mod: client rendering/particles (RenderType moves), fluid interaction hooks, and entity attachment / network payload APIs. Keep each behind small seams so guards stay local.
  • MC3DPrint guard lore applies verbatim: single-hop replacement pairs; twice-moving APIs need if/elif guard chains; never nest block-comment guards; never open a guard with bare // lines; reset active node → 1.21.1 before every commit.
  • scripts/build-all.sh → one mcwaterslides-<ver>-neoforge-<node>.jar per node into dist/; release.yml runs it on GitHub Release publish and attaches all jars.
